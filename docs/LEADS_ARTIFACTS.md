# Leads Artifacts: Origin, Purpose, and Usage

This document explains the lead-related files generated by the autonomous worker run on 2026-03-09 and how to use them safely without deleting artifacts.

## What was generated and why

During a `tech-lead` task chain in the universal worker, the agent attempted to:

1. Define a lead data format.
2. Implement an API endpoint for listing leads.
3. Implement a `list_leads` server tool.
4. Write a lead-listing API spec.

These artifacts were generated as implementation drafts and then partially hardened for Supabase usage.

## File-by-file status

### 1) `lead_listing_api_spec.md`

- Status: Draft specification.
- Purpose: Documents query params, filtering, pagination, and response shape for a lead listing API.
- Use it for: Product/API design alignment before final backend implementation.

### 2) `pages/api/leads/index.js`

- Status: Template endpoint (Supabase-based).
- Purpose: Ready handler for `GET` lead listing with `status`, `q`, `page`, `limit`.
- Runtime note: This repo is Vite-first, so this file is not automatically exposed unless you run it in a server framework that supports `pages/api` routes.

### 3) `src/pages/api/leads/index.js`

- Status: Mirror template for compatibility.
- Purpose: Keeps a consistent endpoint draft in `src/` where some autonomous tasks attempted writes.
- Runtime note: Also not active by default in the current Vite setup.

### 4) `server/tools/list_leads.js`

- Status: Practical CLI tool (Supabase).
- Purpose: Query `crm_leads` directly from the terminal.
- This is the recommended current usage path.

## Recommended current usage

Use the CLI tool for real data access now:

```bash
npm run leads:list -- --limit 20
npm run leads:list -- --status new
npm run leads:list -- --q hotel
npm run leads:list -- --stale_days 14
```

Supported filters:

- `--limit <1..200>`
- `--status <value>`
- `--q <text>` (matches first/last/company/email)
- `--stale_days <days>`

## Required environment variables

For both the endpoint template and CLI script:

- `VITE_SUPABASE_URL`
- `VITE_SUPABASE_SERVICE_ROLE_KEY`

## Integration path (if you want this live as API)

1. Choose the backend runtime (`Next.js API routes`, `Express`, `Supabase Edge Function`, etc.).
2. Move/adapt the handler logic from `pages/api/leads/index.js` into that runtime.
3. Keep `server/tools/list_leads.js` for operational/QA queries.
